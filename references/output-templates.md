# 输出骨架：按问题裁剪

选择最短但足以让用户理解、核验和后续行动的结构。`assets/` 中提供可编辑起点；不要为了填模板而制造状态、目标价、仓位或情景。只有用户要求长期记忆时，才将可复用的资料与结论整合进 LLM Wiki。报告中的收入、利润、现金流、资产负债表和关键公司行为引用公司或交易所的 PDF/原文；TuShare 只作为市场数据、时间线或进一步阅读的线索。

把内部证据记录当作底稿，而不是正文版式：先说明关键判断或发现，再交代依据、解释、反方情况和限制。用来源归属、时间边界、因果衔接和条件语态使证据状态自然可辨；逐项核验、横向比较或消除歧义时，显式标签和表格更合适。内部字段可以服务追溯，但不必逐句成为正文的句首或小标题。

## 轻量回答

适用于数据核验、价格/新闻归因、单一问题或信息不足的场景：

```text
问题与 as_of
直接回答
关键证据与来源
反方解释 / 不确定性
下一步或需要补充的信息
```

## 候选比较或研究计划

适用于发现、初筛或尚不应给出行动结论的情况：

```text
范围、筛选逻辑和遗漏风险
候选及各自值得研究的原因
最关键的未知项和验证路径
优先级的理由（不是投资评级）
下一轮需要的数据或用户选择
```

## 投资备忘录

适用于用户明确询问是否买入、持有、加减仓或比较替代方案：

```text
问题、as_of、持有期和已知约束
当前条件化判断
长期逻辑与最强反方证据
当前估值、价格、事件或组合条件
与现金/替代标的的比较（如适用）
风险、证伪条件和复核触发器
来源、数据口径与关键缺口
```

如果缺少持仓、风险预算或价格输入，明确限制，不要填入精确仓位或交易指令。

## 深度报告

在复杂、金额重大、争议较大或用户要求深度研究时扩展为：

1. 研究问题、范围、`as_of` 和当前结论。
2. 核心假设、商业/财务/竞争证据与反证。
3. 与问题相关的行业、宏观、事件、估值、价格和组合分析。
4. 情景或条件化路径，以及关键变量敏感性。
5. 风险、证伪、数据缺口、保留分歧和后续验证计划。
6. 来源与必要的数据附录。

可按标的重排这些章节。例如，政策驱动行业应扩大政策传导；周期股应扩大供需与资产负债表；短期归因则可以完全不写长期章节。

## 按需整合到 LLM Wiki

用户要求复用或更新持续研究时，才读取相关 Wiki 页面，并将本轮新增的原始资料、关键事实、假设、条件化结论、反证和验证触发器整合回对应领域。原始资料使用 `raw/<domain>/<subject>/<YYYY-MM-DD>/<内容明确的文件名>`；综合内容使用 `wiki/<domain>/<subject>/<内容页面>.md`，再按需更新 `index.md` 与 `log.md`。归档原始资料本身不要求开启 Wiki 综合层。详见 `references/wiki-memory.md`。
